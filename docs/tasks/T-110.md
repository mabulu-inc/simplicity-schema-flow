## Description

RLS policies are dropped and recreated on every plan because `policyChanged()` does a strict string comparison on the USING clause. PostgreSQL normalizes expressions via `pg_get_expr()`, so the introspected value never matches the YAML definition — even immediately after `schema-flow run` creates the policy.

For example:

- YAML: `tenant_id = current_setting('app.current_tenant')::uuid`
- DB: `(tenant_id = (current_setting('app.current_tenant'::text))::uuid)`

PostgreSQL adds outer parentheses, inner parentheses around function calls, and explicit `::text` casts on string literal arguments.

### Implementation

- Normalize both desired and introspected USING/CHECK expressions before comparison in `policyChanged()`.
- The recommended approach is to round-trip the desired expression through PostgreSQL (create a temporary policy in a rolled-back transaction, read back the normalized form via `pg_get_expr()`) so PostgreSQL handles all normalization. Evaluate simpler approaches first if they are sufficient.
- After a successful `schema-flow run`, a subsequent `schema-flow plan` must report 0 operations for unchanged RLS policies.
