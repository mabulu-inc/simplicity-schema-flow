# T-103: Auto-migrate \_simplicity schema to \_smplcty_schema_flow

- **Status**: TODO
- **Milestone**: 30 — Internal Schema Rename
- **Depends**: T-102
- **PRD Reference**: §2
- **Complexity**: standard

## Description

Users upgrading from a prior version may have an existing `_simplicity` schema containing history, snapshots, and expand_state data. The tool should automatically rename it to `_smplcty_schema_flow` on first run so no data is lost and no manual intervention is required.

## Implementation

In `src/core/tracker.ts`, add a migration step to `ensureHistoryTable()` that runs **before** `CREATE SCHEMA IF NOT EXISTS _smplcty_schema_flow`:

1. Check if `_simplicity` schema exists (query `pg_namespace`)
2. If it exists **and** `_smplcty_schema_flow` does not exist, run `ALTER SCHEMA _simplicity RENAME TO _smplcty_schema_flow`
3. If both exist, this is an edge case — log a warning and leave `_simplicity` alone (user can manually reconcile)
4. Proceed with normal `CREATE SCHEMA IF NOT EXISTS _smplcty_schema_flow`

Also rename any dual-write triggers/functions that use the old `_simplicity_dw_` prefix to `_smplcty_sf_dw_` — query `pg_trigger` and `pg_proc` for objects matching the old prefix pattern and rename them.

## Produces

- `src/core/tracker.ts` — migration logic in `ensureHistoryTable()`
- `src/core/__tests__/tracker.test.ts` — tests for the migration path
- Test: fresh install (no prior schema) still works
- Test: existing `_simplicity` schema is renamed and all data preserved
- Test: both schemas exist triggers a warning

## Visibility

The migration is **not** a planned operation — it runs as internal infrastructure setup in phase 0, same as `CREATE SCHEMA IF NOT EXISTS`. It will not appear in `plan`, `drift`, or `--dry-run` output. However, when the rename actually executes, it must log a clear info-level message (e.g., `Migrated internal schema: _simplicity → _smplcty_schema_flow`).

## Acceptance Criteria

- Existing `_simplicity` schema with history/snapshots/expand_state is seamlessly renamed
- All data (history entries, snapshots, expand state) is preserved after rename
- Fresh installs are unaffected
- Dual-write triggers/functions with `_simplicity_dw_` prefix are renamed to `_smplcty_sf_dw_`
- `pnpm check` passes
