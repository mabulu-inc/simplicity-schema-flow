## Description

T-115 attempted to fix table-level grant drift but the fix is incomplete. The bug still reproduces with `privileges: [DELETE, INSERT, SELECT, UPDATE]` on v0.3.6.

Root cause: PostgreSQL introspects column-level grants as **one grant per privilege per column** (e.g., a separate entry for SELECT on col `id`, INSERT on col `id`, etc.). The T-115 fix builds a lookup key from the actual grant's full privilege list, but each introspected grant has only a single privilege. So the key `app_user:SELECT` never matches the desired key `app_user:DELETE,INSERT,SELECT,UPDATE`.

The fix needs to check whether all of an actual grant's individual per-column privileges are a subset of a table-level desired grant's privileges for the same grantee, rather than requiring an exact key match.

The RED test must use `privileges: [DELETE, INSERT, SELECT, UPDATE]` (the exact combination from the bug report) with a multi-column table — not the simplified `[SELECT]` or `[SELECT, INSERT, UPDATE]` cases that T-115 tested. The T-115 tests passed because they never exercised the per-privilege introspection mismatch.

Close GitHub issue #13 after task completion.
