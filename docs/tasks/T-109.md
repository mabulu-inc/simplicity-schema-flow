## Description

v0.3.0 fixed the original unique constraint re-planning issue (#4) for most cases, but at least one case still produces phantom operations: when a unique constraint and its backing index both exist in the database, `schema-flow plan` still emits `add_index` + `add_unique_constraint` operations.

The T-107 fix (excluding constraint-backed indexes via `pg_constraint` join in `getIndexes`) may not be matching this particular constraint during introspection, causing the planner to think neither the index nor the constraint exists.

3. Comment on GitHub issue #7 with findings (confirmed/invalid, root cause, plan of action).
4. Then proceed with the full dev cycle (RED → GREEN → VERIFY).

### Implementation

- Investigate and fix the introspection gap that causes this specific unique constraint pattern to be missed.
- After a successful `schema-flow run`, a subsequent `schema-flow plan` must report 0 operations for unchanged unique constraints (including when both constraint and backing index exist in the database).
