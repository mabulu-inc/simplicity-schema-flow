# T-087: Use CREATE ROLE IF NOT EXISTS in role pipeline

- **Status**: TODO
- **Milestone**: 28 — Idempotent Pipeline Operations
- **Depends**: —
- **PRD Reference**: §6.1 (pipeline step: Roles)

## Description

The `runAll` pipeline uses `CREATE ROLE` without `IF NOT EXISTS` when applying role YAML files. This causes `42710` (duplicate_object) errors when the pipeline runs against a Postgres cluster where the roles already exist — for example, when multiple test suites call bootstrap on different databases in the same cluster.

## Current state

- Role YAML files define roles (e.g., `anon`, `app_admin`, `app_editor`, `app_viewer`, `authenticator`)
- The pipeline emits `CREATE ROLE <name>` which fails if the role already exists at the cluster level
- Postgres roles are cluster-wide, not per-database, so running the pipeline on a second database in the same cluster triggers the error

## Reproduction

In the `simplicity-admin` CI, the `db` package tests create roles via bootstrap, then the `auth` package tests call bootstrap on a fresh test database — but the roles already exist at the cluster level, causing the pipeline to fail with error code `42710`.

## Fix

Change the role creation SQL to be idempotent. Either:

1. Use `CREATE ROLE IF NOT EXISTS` (Postgres 16+), or
2. Use a `DO` block: `DO $$ BEGIN IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = '...') THEN CREATE ROLE ...; END IF; END $$`

Option 2 is safer for broader Postgres version support.

## Produces

- Updated role pipeline step to use idempotent role creation
- Test verifying that running the pipeline twice on the same cluster does not error on role creation

## AC

- Running the pipeline twice against the same Postgres cluster (same or different databases) does not produce `42710` errors
- Existing role properties (login, inherit, membership) are still applied correctly
- All existing tests continue to pass
- `pnpm check` passes
