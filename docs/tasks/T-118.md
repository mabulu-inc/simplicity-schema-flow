## Completion Notes

Added `normalizeCheckExpression()` to both `drift/index.ts` and `planner/index.ts` that strips redundant `::character varying::text` → `::character varying` and `]::text[]` → `]` casts before comparing check constraint expressions. PostgreSQL's `pg_get_constraintdef()` distributes array-level `::text[]` casts into per-element `::character varying::text` suffixes, causing false drift positives on every run. 3 tests: drift normalization, planner normalization, genuine difference still detected.

## Description

`schema-flow drift` reports every check constraint with varchar `IN (...)` expressions as `[different]` on every run, even immediately after a clean apply. The expressions are semantically identical — PostgreSQL adds a redundant `::text` cast when serializing via `pg_get_constraintdef()`:

- YAML/applied: `'source'::character varying`
- Introspected: `'source'::character varying::text`

The drift comparison (`driftChecks` and `diffChecks` in the planner) uses strict string equality with no normalization. The codebase already has `normalizeWhitespace()` and `normalizeTypeName()` for other comparisons, but check expressions bypass all normalization.

Add a `normalizeCheckExpression()` function that strips redundant trailing `::text` casts from `::character varying::text` sequences (since `::character varying` already produces text). Apply it in both `driftChecks` (drift/index.ts:532) and `diffChecks` (planner/index.ts:1169) before comparing.

The RED test must use the exact expression pattern from the bug report — `IN (...)` with varchar values — not simple integer comparisons. The T-116 tests missed this bug because they only tested `amount > 0` style expressions that PostgreSQL round-trips without cast changes.

Close GitHub issue #16 after task completion.
