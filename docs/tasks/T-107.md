## Description

Two related bugs:

1. **Introspection does not recognize existing unique constraints.** The live database read sees the backing index in `pg_indexes` but does not cross-reference `pg_constraint` (`contype = 'u'`) to detect that a unique constraint already exists. This causes phantom `drop_index` + `add_unique_constraint` on every plan.

2. **Drop step emits `DROP INDEX` instead of `ALTER TABLE ... DROP CONSTRAINT`.** PostgreSQL does not allow dropping an index that backs a constraint — it requires `ALTER TABLE <table> DROP CONSTRAINT <name>` instead.

### Ralph must first act as product manager

1. Verify both sub-issues by reading the introspection and planner code.
2. Confirm the constraint-backed index distinction is not currently handled.

4. Then proceed with the full dev cycle (RED → GREEN → VERIFY).

### Implementation

- Update introspection to cross-reference `pg_constraint` (contype = 'u') with `pg_indexes` so unique constraints are recognized as constraints, not bare indexes.
- Update the planner to not emit `drop_index` + `add_unique_constraint` when the constraint already exists and matches.
- Update the executor to emit `ALTER TABLE ... DROP CONSTRAINT` instead of `DROP INDEX` for constraint-backed indexes.
- After a successful run, `schema-flow plan` must report 0 operations for unchanged unique constraints.
