## Description

The planner emits `CREATE UNIQUE INDEX CONCURRENTLY "<name>" ...` for unique constraints without an idempotency guard. If the index already exists, the statement fails with `42P07` (duplicate_table — PG uses this code for duplicate indexes too).

## Current state

- `src/planner/index.ts` line 1177: `CREATE UNIQUE INDEX CONCURRENTLY "${ucName}" ON "${pgSchema}"."${table}" (${cols})`
- `src/planner/index.ts` line 1185: `ALTER TABLE ... ADD CONSTRAINT "${ucName}" UNIQUE USING INDEX "${ucName}"`
- PostgreSQL supports `CREATE INDEX CONCURRENTLY IF NOT EXISTS` (PG 9.5+)
- `reindexInvalid()` in the executor already handles cleanup of invalid indexes from failed CONCURRENTLY operations

## Fix

Add `IF NOT EXISTS` to the `CREATE UNIQUE INDEX CONCURRENTLY` statement:

```sql
CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS "<name>" ON "<schema>"."<table>" (...)
```

For the `ALTER TABLE ADD CONSTRAINT ... USING INDEX`, guard with a `pg_constraint` catalog check:

```sql
DO $$ BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_constraint WHERE conname = '<name>') THEN
    ALTER TABLE "<schema>"."<table>" ADD CONSTRAINT "<name>" UNIQUE USING INDEX "<name>";
  END IF;
END $$
```
