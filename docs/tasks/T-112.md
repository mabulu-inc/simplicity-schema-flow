# T-112: Fix drift false positive for single-column unique constraints

- **Status**: TODO
- **Milestone**: 35 — Bug Fixes: Drift False Positives
- **Depends**: none
- **PRD Reference**: §6
- **GitHub Issue**: #9
- **Complexity**: standard

## Description

When a column has `unique: true` in the YAML definition, PostgreSQL creates a `UNIQUE CONSTRAINT` (which implicitly creates a backing index). The drift check then reports two false positives:

1. **Index missing from DB** — `diffTableIndexes` in `src/drift/index.ts` filters out constraint-backing indexes via `!i.constraint_type`, so the backing index is excluded from comparison.
2. **Constraint extra in DB / missing from YAML** — `diffUniqueConstraints` in `src/drift/index.ts` only keeps multi-column constraints (`if (cols.length <= 1) existingUcMap.delete(name)`), so single-column unique constraints are discarded.

Single-column unique constraints fall into a gap between these two checks and are always reported as drifted.

The fix should either:

- Include single-column unique constraints in `diffUniqueConstraints`, or
- Recognize constraint-backing indexes for `unique: true` columns as expected in `diffTableIndexes`

After `schema-flow run` on a fresh database, `schema-flow drift` must report zero drift for tables with single-column `unique: true` columns.

Close GitHub issue #9 after task completion.
