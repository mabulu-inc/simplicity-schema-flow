# T-114: Fix drift false positive when column unique: true overlaps explicit index

- **Status**: TODO

- **GitHub Issue**: #11
- **Complexity**: standard

## Description

When a YAML table defines both `unique: true` on a column and an explicit `indexes` entry with the same name (e.g. `roles_name_key`), `schema-flow drift` reports three false positives after a clean apply:

1. `index: roles_name_key [missing_in_db]` — the explicit index entry is not found because PostgreSQL created a constraint-backed index (which is filtered out by `!i.constraint_type`)
2. `constraint: roles.roles_name_key [missing_in_yaml]` — the constraint exists in DB but is excluded by the column-level unique logic
3. `seed: roles [different]` — seed data comparison still fails despite T-113 fix (likely a related edge case)

The root cause is that T-112 handled single-column `unique: true` constraints in isolation, but did not account for the case where the user also declares a redundant explicit index with the same name. The drift check needs to recognize that a constraint-backed index satisfies an explicit index declaration when the names match.

After `schema-flow apply` on a fresh database with the reproduction YAML from issue #11, `schema-flow drift` must report zero drift.

Close GitHub issue #11 after task completion.
